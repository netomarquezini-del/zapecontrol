**Epic:** 5 — SDR Station
**Status:** Draft
**Executor:** @dev

## Objetivo
Implementar dashboard completo de métricas de performance dos SDRs.

## User Story
> Como gestor, quero um dashboard com métricas de performance dos SDRs, para tomar decisões baseadas em dados.

## Acceptance Criteria

- [ ] API `GET /api/sdr/metrics` — endpoint consolidado de métricas com filtros (período, sdr_id)
- [ ] API `GET /api/sdr/metrics/heatmap` — dados de heatmap (conexões por hora/dia da semana)
- [ ] API `GET /api/sdr/metrics/funnel` — conversão por etapa do pipeline
- [ ] API `GET /api/sdr/metrics/channels` — métricas por canal (telefone vs WhatsApp vs Instagram)
- [ ] Página `/comercial/sdr-station/metricas` com período selecionável (hoje, 7 dias, 30 dias, custom)
- [ ] KPIs principais em cards:
  - Taxa de conexão (chamadas atendidas / total discado)
  - Chamadas por hora
  - Tempo médio de conversa
  - Custo por conexão (estimado baseado em minutos Twilio)
- [ ] Gráfico de funil: conversão por etapa do pipeline (Novo → Tentativa → ... → Agendado)
- [ ] Heatmap: melhor horário para ligar (grid hora x dia da semana, cor = taxa de conexão)
- [ ] Ranking comparativo entre SDRs (tabela com KPIs por SDR)
- [ ] Taxa de no-show em agendamentos (agendados vs realizados)
- [ ] Speed to lead: tempo médio entre lead entrar e primeira tentativa de contato
- [ ] Métricas por canal:
  - WhatsApp: taxa de resposta, tempo médio de resposta
  - Instagram: taxa de resposta, tempo médio de resposta
  - Telefone: taxa de conexão, duração média
- [ ] Gráficos interativos (hover com detalhes)
- [ ] Export básico dos dados (CSV)
- [ ] Dashboard carrega em < 3 segundos

## Integration Verification
- IV1: Queries performam bem com 3+ meses de dados
- IV2: Dashboard carrega em < 3 segundos
- IV3: Dados consistentes com registros nas tabelas de calls, messages e schedules

## Notas Técnicas
- Charts: usar Recharts (já pode estar no projeto) ou Chart.js
- Queries: usar views materializadas ou queries otimizadas com índices adequados
- Heatmap: agrupar chamadas atendidas por hour_of_day e day_of_week
- Speed to lead: diff entre sdr_leads.created_at e primeiro sdr_calls.created_at do lead
- Custo por conexão: (total_minutes * custo_por_minuto_twilio) / total_conexões
- Período custom: date range picker component
- Para performance com volume grande: considerar aggregation tables atualizadas por cron

## Testing Guidance
- **Approach:** Integration + Performance
- KPIs: verificar cálculos com dados mock (taxa conexão, chamadas/hora, etc.)
- Funil: conversão por etapa com dados reais vs esperado
- Heatmap: dados agrupados corretamente por hora/dia
- Filtros: período (hoje, 7d, 30d, custom) retorna dados corretos
- Performance: dashboard carrega em < 3 segundos com 3 meses de dados
- Métricas por canal: WhatsApp, Instagram, Telefone calculados separadamente
- Export CSV: arquivo gerado com dados corretos
- Ranking SDR: comparativo funciona com 2+ SDRs
